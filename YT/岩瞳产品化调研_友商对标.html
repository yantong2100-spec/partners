我们现在在哪 · 路桥线代码实测

基于 2026-07-21 对 E:\allProjects\luqiao 仓库的实际检视，非自我陈述

一句话现状

路桥线已实现「图片/视频输入 → SAM 分割/自动检测 → 面积量化 → VLM 分析 → JTG 评分 → docx/PDF 报告」的完整端到端流程，有约14座真实桥梁数据和24组已生成报告（含苏交科 POC），软著5项+专利3项材料齐备；但产品层（政府版/企业版 SDK）目录为空壳，无用户体系、无计费、无对外 API，仍是单机演示应用形态。git 共15次提交（2026-01-15 至 2026-04-20），4月底仍有报告产出，开发节奏为间歇式。

已经有什么（实测确认）

  • 端到端流程跑通：交互点选分割、自动批量检测、视频逐帧追踪、比例标定面积量化、JTG/T H21-2011 评分（SPCI/SBCI/BDCI/Dr）、docx/PDF/Excel/CSV 报告导出
  • 前端存在：Gradio Web UI（app.py 约3900行），功能可用，定位是演示/内部工具
  • 一键部署：make install 双模式。Cloud 模式（fal.ai SAM3+云端 VLM，磁盘<200MB，普通笔记本可跑）；Local 模式（SAM2.1 本地推理+Ollama，需32GB+内存）
  • 真实数据与交付物：real_data 下约14座真实桥梁（三叉溪、东溪河特大桥含点云、苏交科POC拱桥等），reports 下24组检测/对比报告，最近一批 2026-04-28
  • 知识产权：软著材料5项（整体系统/SAM3分割引擎/面积量化/报告生成/视频追踪，各配演示程序）+ 专利材料3项（多模态融合检测/面积量化方法/跨帧追踪），申报中
  • 报告工程沉淀：yantong-skills 仓库有桥梁定检、隧道定检各一套 builder+reviewer 双闸门报告生成 skill，已在真实项目使用

还缺什么（诚实评估）

  • 产品层为空：products/ 下政府版（gov_solution）与企业版 SDK（enterprise_sdk）仅有空的 __init__.py，「双轨战略」停留在文档层面
  • 无商业化基础设施：无用户注册/登录、无权限体系、无计费模块、无对外 API 接口；客户无法自助上手，只能由岩瞳人员操作演示
  • Skills 不是产品：报告生成 skill 是 Claude Code 内部工作流，客户不可直接使用
  • 技术阻塞：SAM 3 为 CUDA-only（Mac 本地不可用，靠 fal.ai 云端补位）；Gemini 免费层限流影响批量分析
  • 定价未经验证：docs/定价策略.md 已推演客户分类、三档功能分层、四种计费模式，但均为内部推演，没有一单实际验证

检视方式：目录树遍历、README/CLAUDE.md/docs 阅读、核心模块代码抽样（app.py、auto_detector.py 1218行、jtg_report_generator.py 1165行等）、git log、real_data 与 reports 产出物核对。

  • API 开放做得好的公司都把「数据读取 API」和「控制/写入 API」分开设计（Signify 是最清晰的模板）
